# Sidekiq Worker Generator

**Scale:** at scale — once Sidekiq is in the stack and you're writing background jobs

Scaffolds a Sidekiq worker with the correct conventions: scalar arguments only, service delegation, explicit queue and retry options, and a `rescue ActiveRecord::RecordNotFound` guard. Includes a spec stub.

---

## When to use it

Any time you're adding a new Sidekiq worker. The generated structure enforces two conventions that are easy to get wrong:

- **Scalar args only** — passing ActiveRecord objects or hashes to `perform` causes serialization issues that are hard to debug. The scaffold uses primitive IDs and looks up records inside the worker.
- **RecordNotFound rescue** — workers that look up records by ID should rescue and discard rather than retry infinitely on a deleted record.

---

## Prerequisites

- Sidekiq in your Gemfile
- `app/services/` with `ApplicationService` (generated by `/rails-boilerplate`)
- FactoryBot set up for the spec stub (generated by `/rails-boilerplate`)

---

## Installation

Copy the skill file into your project's `.claude/commands/` directory:

```bash
mkdir -p .claude/commands
cp worker-generator.md .claude/commands/worker-generator.md
```

---

## Usage

In a Claude Code session:

```
/worker-generator
```

Claude asks four questions, then generates the worker and spec in one pass.

---

## What gets generated

```
app/workers/
  <name>_worker.rb          # Worker with sidekiq_options, scalar args, service delegation, RecordNotFound rescue

spec/workers/
  <name>_worker_spec.rb     # Spec: service called with correct args, no raise on RecordNotFound
```

---

## Example output

For a worker named `SendWelcomeEmail`, queue `mailers`, retry `3`, primary arg `user_id`:

```ruby
# app/workers/send_welcome_email_worker.rb

class SendWelcomeEmailWorker
  include Sidekiq::Worker
  include Loggable

  sidekiq_options queue: :mailers, retry: 3

  def perform(user_id)
    user = User.find(user_id)
  end
end
```

```ruby
# spec/workers/send_welcome_email_worker_spec.rb

require "rails_helper"

RSpec.describe SendWelcomeEmailWorker, type: :worker do
  describe "#perform" do
    let(:user) { create(:user) }

    it "performs without error" do
      expect { described_class.new.perform(user.id) }.not_to raise_error
    end
  end
end
```

---

## Conventions enforced

- Workers live in `app/workers/`, named `<Action>Worker`
- `perform` arguments are scalar only — integers, strings, booleans. Never ActiveRecord objects or hashes
- `sidekiq_options` always explicit — queue and retry set per worker, never relying on defaults
- `ActiveRecord::RecordNotFound` is not rescued by default — let Sidekiq retry and surface the failure. Add rescue-and-discard explicitly when a missing record is expected and unrecoverable

---

## After generating

- Rename the service call if the service name doesn't match the convention
- Add the worker to the appropriate queue in your Sidekiq config if the queue is new
- Run `bundle exec rspec spec/workers/<name>_worker_spec.rb` to confirm the spec passes

## At scale

Once you have multiple workers, add `app/workers/CLAUDE.md` to your project. Claude Code loads it automatically when working in that directory, keeping worker conventions in session context without polluting the root CLAUDE.md.

A template is available at [`claude-md-hierarchy/workers/CLAUDE.md`](../../claude-md-hierarchy/workers/CLAUDE.md) — copy it into your project and fill in the queue and retry markers.
